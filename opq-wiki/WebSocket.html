# 正向/反向WebSocket介绍

  • 框架内置了SoocketIO客户端和服务端以及WebSocket客户端和服务端,在配合WebApi很容易开发出自己熟悉语言的第三方框架,SocketIO是对Websocket的一种实现,因为内置了一些事件集合,管理起来也比较方便,内置心跳等, SocketIO协议类似[CMD Playloads]有兴趣的小伙伴可以自己研究一下

# 框架作为Websocket服务端:

  • 默认的Websocket服务端和WebApi同一个地址绑定 也就是说是和用户配置的CoreConf.conf文件里的 Port = "0.0.0.0:8888"监听地址绑定的 自然Websocket服务器地址就是0.0.0.0:8888 暂时不支持wss协议
  • 链接地址 ws://0.0.0.0:8888/ws

# 框架作为WebSocket客户端:

  • 默认是不启用的通过配置CoreConf.conf文件里的 SocketIOUrl = "ws://127.0.0.1:8081/ws" 来启用

  • 这时框架会主动链接你的WebSocket的服务器并且带有断线重连功能

# 框架作为SocketIO服务端:

框架提供了3种Event:OnGroupMsgs,OnFriendMsgs,OnEvents

  • OnGroupMsgs 收到群消息会推送到OnGroupMsgs

  • OnFriendMsgs 收到好友消息会推送到OnFriendMsgs

  • OnEvents 收到QQ的各种事件 如有人进群退群,网络变化事件都会推送到OnEvents

  • 默认的SocketIO服务端和WebApi同一个地址绑定 也就是说是和用户配置的CoreConf.conf文件里的 Port = "0.0.0.0:8888"监听地址绑定的 自然SocketIO服务器地址就是0.0.0.0:8888 暂时不支持wss协议

# 框架作为SocketIO客户端:

  • 默认是不启用的通过配置CoreConf.conf文件里的 SocketIOUrl = "ws://127.0.0.1:8081/socket.io/?EIO=3&transport=websocket" 来启用

  • 当然也可以连接你的原生Websocket,这时你的服务器收到的内容就是类似[CMD Playloads]这个了,并且你还要实现一下心跳ping/pong不然不会长时间连接,还是建议连接SocketIO服务器哦

  • 这个功能为内网用户提供了方便,免去了端口映射,安全性也增加了
